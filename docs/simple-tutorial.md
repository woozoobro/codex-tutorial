## 플젝 생성

그럼 shh-eat 컨셉의 새로운 프로젝트를 하나 생성해보자. 플젝 이름은 웬만하면 영어로, 공백없이 하는게 좋음.
(처음부터 시작을 클릭 하고 플젝 이름 만들어서 생성하는거 보여주기.)
그럼 플젝이 생성되는데 내 컴퓨터 어디서 파일들이 생성되는지 보고 싶으면 우클릭, 파인더에서 보기를 해주면 됨.
앞으로 작업할 파일들은 이 위치에 저장될거다.

이제 채팅을 해볼건데
코덱스에서 프로젝트에서의 채팅 세션들, 채팅창들을 '스레드'라고 표현함.
그럼 우리가 일할 땐 주로 스레드에서 하게 되는데.

우리 이 shh-eat이라는 플젝에대해서 어떤 내용인지 지금처럼 매번, 새로운 채팅창을 열때마다 설명하면 불편하지 않음?
그래서 AGENTS.md라는 걸 만들어서 '이 프로젝트는 어떤 프로젝트에요' 를 매번 입력해도 되지 않게 해줌.

## AGENTS.md 작성법

단순함. 걍 만들어달라하면 됨.
아이데이션한 내용 바탕으로.

```
---

이 내용 바탕으로 AGENTS.md 현재 플젝에 만들어보자.
```

대략적인 프로젝트에 대한 맥락 정보, context가 이제 AGENTS.md에 담겼다.
이런 AGENTS.md는 AI 에이전트 시대의 표준이니 꼭 기억해두셔야함.

이렇게 만든 AGENTS.md는 이제 작업하면서 계속 수정해 나가면 됨.
우리 플젝 방향성이 언제든 바뀔 수 있잖음.

```
/OpenAI Docs 현재 AGENTS.md best practice로 작성된거 같음? 체크점
```

나중에 배울거긴 한데 스킬이라는 게 있어. 

openai docs 스킬을 활용해서 AGENTS.md best practice로 만들어졌는지 체크해달라고 하면
알아서 공식 문서 읽고 어떻게 AGENTS.md 만들어야할지 알려줌.

일단 이렇게 입력.

```
아이데이션,기획 다끝나고 나면 나중에 랜딩까지 만들어볼라구.

일단 지금 문서 최대한 간결하게 변경해주라.
```

자 이렇게 AGENTS.md를 만들고 우리 프로젝트가 어떤 내용인지를 작성해두면,
어떤 채팅창, 스레드를 시작하더라도 이 내용은 이미 채팅창에 올라가 있는 상태다.

## github

이 프로젝트를 구글 드라이브 같은 곳에 업로드 해두고, 협업을 하면 편하다.

## 웹에서 시장 조사해보기

자 이제 프로젝트 기본 맥락도 잡았고, github에 올릴 준비도 됐다.
다음으로 해볼 건 웹 리서치.

지금까지는 우리가 상상한 아이디어를 바탕으로 브랜드 컨셉을 잡은 거임.
근데 실제 업무에서는 여기서 끝내면 안 됨.

이 아이템이 진짜 시장에서 말이 되는지,
비슷한 제품이나 서비스는 이미 있는지,
직장인 간식이나 오피스 스낵 시장에서는 어떤 메시지가 쓰이는지
한번 찾아봐야 함.

codex한테 웹에서 검색해서 시장 조사하고, 그 내용을 docs 폴더에 문서로 저장해달라고 해보자.

(새 스레드에서 프롬프트 입력)
```
Shh-eat 브랜드를 더 구체화하기 위해 웹에서 시장 조사를 해줘.

조사 주제:
- 오피스 스낵 / 직장인 간식 시장 트렌드
- 회의 중 먹기 좋은 간식, 책상에서 먹기 좋은 간식 관련 니즈
- 소음이 적고 가루가 덜 묻는 간식이나 포장 사례
- 유사한 D2C 스낵 브랜드 또는 오피스 간식 구독 서비스
- Shh-eat이 차별화할 수 있는 포지션

요구사항:
- 실제 웹에서 찾은 정보만 사용해줘
- 출처 링크를 같이 남겨줘
- 한국어 자료들을 참고해줘
- 근거가 약한 내용은 약하다고 표시해줘
- 결과는 docs 폴더에 md파일로 정리해줘
```

이렇게 하면 codex가 단순히 말로만 답하는 게 아니라,
프로젝트 안에 파일을 만들어서 결과를 남겨준다.
여기서 문서를 클릭하면 어떤 내용을 얘가 리서치했는지 볼 수 있다.

리서치가 끝났으면 이 내용을 바탕으로 브랜드 전략 문서를 하나 더 만들어보자.

(프롬프트 입력)
```
방금 만든 시장 조사 문서를 참고해서
Shh-eat의 브랜드 기획 방향을 다시 정리해줘.
docs 폴더에 만들면 됨.

포함할 내용:
- 브랜드 한 줄 정의
- 핵심 타깃
- 고객 문제
- 제품이 해결하는 방식
- 경쟁 제품과의 차별점
- 랜딩 페이지에서 강조해야 할 메시지
- 피해야 할 과장 표현
```

이제 프로젝트 안에는 단순한 채팅 기록이 아니라,
`docs/research/market-research.md`,
`docs/brand/brand-strategy.md`
같은 실제 업무 문서들이 쌓이기 시작한다.

이것도 커밋하고 푸시합시다.
커밋에 ... 표시 클릭하고 푸시 하면 한번에 커밋 이후에 푸시까지 알아서해줍니다.

이렇게 작업하다가 중간중간 AGENTS.md를 업데이트 해줘! 라고 해도됨. 큰변경사항 생기게 되면.


## 새 스레드에서 파일 멘션해서 작업하기

자 이제 문서들이 좀 쌓였으니까,
새로운 작업은 새 스레드에서 시작해보자.

이번에는 브랜드 비주얼 방향을 잡고,
이미지 생성으로 제품 목업까지 만들어볼 거임.

여기서 보여줄 건 `@` 멘션.
새 스레드를 열어도, 프로젝트 안에 있는 파일을 `@`로 멘션하면
codex가 그 파일을 참고해서 작업할 수 있다.

AGENTS.md는 프로젝트 전체에 계속 깔리는 기본 맥락이고,
`@` 멘션은 이번 작업에서 특별히 참고해야 하는 파일을 직접 찍어주는 거라고 생각하면 됨.

(새 스레드에서 프롬프트 입력)
```
@docs/research/market-research.md
@docs/brand/brand-strategy.md

이 두 문서를 참고해서 Shh-eat의 브랜드 비주얼 방향을 정리해줘.
결과는 docs 폴더에 md 파일로 저장해줘.
```

이렇게 하면 앞에서 만든 시장 조사, 브랜드 전략 내용을 다시 길게 복붙하지 않아도 된다.
필요한 파일만 멘션해서, 그 내용을 바탕으로 다음 작업을 이어갈 수 있음.


## 이미지 생성으로 제품 목업 만들기

비주얼 방향까지 잡았으면,
이제 실제 제품이 있는 것처럼 보이는 이미지를 만들어보자.

이번에는 두 가지를 만들어보면 됨.

1. 상세페이지 첫 화면에 쓸 제품 패키지 목업
2. SNS 런칭 포스트에 쓸 라이프스타일 이미지

먼저 제품 패키지 목업부터 만들어보자.

/imagegen 이라고 입력하시고
(프롬프트 입력)
```
방금 만든 브랜드 비주얼 방향 문서를 참고해서
Shh-eat의 제품 패키지 목업 이미지를 생성해줘.

이미지 컨셉:
- 회의 중에도 조용히 먹을 수 있는 저소음 오피스 스낵 브랜드
- 브랜드명은 Shh-eat
- 깔끔한 오피스 데스크 위에 놓인 미니 스낵 패키지
- 상세페이지 첫 화면에 사용할 수 있는 제품 중심 이미지
- 차분하고 세련된 D2C 커머스 브랜드 느낌
```

이미지가 생성되면 마음에 드는 방향인지 확인하고,
필요하면 바로 수정 요청하면 된다.

예를 들면 이런 식.

```
좋은데 조금 더 프리미엄 오피스 브랜드처럼 보이게 바꿔줘.
패키지는 더 미니멀하게, 배경은 더 밝고 깨끗하게 해줘.
```

이번에는 SNS에 쓸 수 있는 라이프스타일 이미지를 만들어보자.

(프롬프트 입력)
```
방금 만든 브랜드 비주얼 방향 문서를 참고해서
Shh-eat의 SNS 런칭 포스트에 쓸 라이프스타일 이미지를 생성해줘.

이미지 컨셉:
- 화상회의 중인 직장인의 책상 위에 Shh-eat 스낵이 자연스럽게 놓여 있음
- 조용히 먹을 수 있는 오피스 스낵이라는 메시지가 느껴져야 함
- 노트북, 메모장, 커피, 작은 스낵 패키지가 있는 깔끔한 업무 환경
- 실제 브랜드 콘텐츠처럼 자연스럽게
- 한국 직장인이 봐도 어색하지 않은 분위기
```

여기서 중요한 건 이미지도 그냥 "예쁘게 만들어줘"라고 하지 않는 것.
앞에서 만든 리서치 문서, 브랜드 전략 문서, 비주얼 방향 문서를 바탕으로 만들면
결과물이 훨씬 일관되게 나온다.

다 만들고 나면 docs 폴더에 이미지 저장해달라 하면 됨.

이제 프로젝트에는 문서뿐 아니라 브랜드 이미지 시안까지 생겼다.
이것도 필요한 파일들 커밋하고 푸시합시다.


## Gmail 플러그인으로 메일 초안 쓰기

자 이제 시장 조사 문서, 브랜드 전략 문서, 비주얼 방향 문서,
제품 목업 이미지까지 만들었다.

실제 업무에서는 여기서 끝이 아니라,
이걸 팀장님이나 팀원한테 공유하고 다음 미팅에서 논의해야 함.

이번에는 Gmail 플러그인을 연결해서
미팅 전에 보낼 메일 초안을 만들어보자.

먼저 왼쪽 사이드바에서 `플러그인`을 클릭한다.
여기서 Gmail을 검색하고, Gmail 플러그인을 Codex에 추가하면 됨.

플러그인을 추가하면 Google 계정 연결 화면이 뜰 수 있다.
메일을 읽거나 초안을 만드는 작업은 내 Gmail 계정에 접근하는 거라서,
처음 사용할 때 권한 승인이 필요하다.

추가가 끝났으면 새 스레드에서 Gmail을 사용해달라고 요청하면 된다.
특정 플러그인을 확실히 쓰고 싶을 때는 `@Gmail`처럼 `@`로 멘션해서 시작하면 됨.

(새 스레드에서 프롬프트 입력)
```
@Gmail
docs 폴더 내용들 확인하고. 
wwoods@kakao.com 여기 우주형 팀장님한테 보낼 gmail 초안 작성해줘.

상황:
- Shh-eat 브랜드 런칭 아이디어를 다음 미팅에서 논의할 예정
- 미팅 전에 시장 조사, 브랜드 전략, 비주얼 방향, 제품 목업 이미지를 간단히 공유하고 싶음
```

여기서 중요한 건 "바로 보내지 말고 초안으로만 작성"이라고 명시하는 것.
메일처럼 실제 외부로 나가는 작업은 사람이 마지막으로 확인하고 보내는 게 좋다.

이렇게 플러그인들을 쓰면 외부의 시스템, 서비스들이랑 연동해서 작업하는 것도 가능하다
Codex가 프로젝트 안에 있는 자료를 바탕으로 실제 업무 도구까지 이어서 사용할 수 있다.


## Plan mode로 큰 작업 계획 세우기

이제 Shh-eat 랜딩 페이지를 실제 웹사이트로 만들어보자.

여기서부터는 작업이 좀 커진다.
Next.js 프로젝트를 셋업해야 하고,
지금까지 만든 문서와 이미지도 참고해야 하고,
로컬에서 실행해서 확인도 해야 한다.

이런 큰 작업은 바로 시작하기보다,
먼저 Plan mode에서 작업 계획을 세우는 게 좋다.

채팅창 아래쪽 권한 설정에서 Plan mode로 바꾸고 시작하면 됨.
Plan mode에서는 바로 파일을 수정하지 않고,
어떤 순서로 작업할지 먼저 계획을 세워준다.

(Plan mode에서 프롬프트 입력)
```
Shh-eat 랜딩 페이지를 Next.js로 만들고 싶어.
현재 shh-eat 폴더를 그대로 쓰고, git은 새로 만들지 않았으면 좋겠어.
지금까지 만든 문서와 이미지(docs에 있음)도 참고해서 진행 계획을 먼저 세워줘.
```

그러면 codex가 대략 이런 것들을 확인하고 계획을 세워줄 거다.

- 현재 프로젝트 구조 확인
- Next.js 셋업 방식 결정
- 랜딩 페이지에 들어갈 섹션 정리
- 참고할 문서와 이미지 확인
- 로컬 실행 방법 정리
- 커밋/푸시 시점 정리

계획을 보고 괜찮으면 이제 진행시키면 된다.

```
좋아. 이 계획대로 진행해줘.
```

nextjs를 설치하기 위한 환경이 구성 안되있으면 어떻게 해야하는지 
이것저것 설치를 할거에요.

이제 설치 다 끝나고나면 뭐 nextjs 개발 서버를 킬까요? 물어보는데 예를 해주시면 됩니다.
이거는 지금 만든 랜딩페이지를 내 로컬환경, 컴퓨터환경에서 보기위해서 개발 서버를 시작하는거라 생각해주시면 되구요.

보통은 npm run dev라고 얘기를 해주면 됩니다.

자 그러고 나면 코덱스가 코덱스 앱 내에서 인터넷 브라우저를 띄울 수가 있어요.
그럼 지금 만들어진 페이지를 혼자서 막 확인을 합니다.

## 배포하기

이제 로컬에서 랜딩 페이지까지 확인했으면,
다른 사람도 볼 수 있게 인터넷에 배포해보자.

배포는 내 컴퓨터에서만 보이던 웹사이트를
팀원이나 고객도 들어가 볼 수 있는 실제 주소로 올리는 거라고 생각하면 됨.

Next.js 프로젝트는 Vercel에 배포하기 좋다.
Vercel은 GitHub 레포랑 연결해두면,
앞으로 코드를 푸시할 때마다 자동으로 새 버전을 배포해준다.

먼저 Vercel 계정을 만들어야 한다.
Vercel은 GitHub 계정으로 로그인하면 편함.

Vercel에 로그인한 뒤,
내 GitHub 레포에 접근할 수 있도록 권한을 승인해준다.

그 다음 Codex에서 Vercel 플러그인을 추가해보자.
왼쪽 사이드바에서 `플러그인`을 클릭하고,
Vercel을 검색해서 추가하면 됨.

처음 사용할 때 Vercel 계정 연결이나 권한 승인이 필요할 수 있다.

(새 스레드에서 프롬프트 입력)
```
@vercel
이 Shh-eat 랜딩 페이지를 Vercel에 배포하고 싶어.
GitHub에 올라가 있는 shh-eat 레포를 사용하면 되고,
배포가 끝나면 접속 가능한 URL을 알려줘.
```

이렇게 요청하면 codex가 Vercel 배포에 필요한 설정을 확인하고,
배포가 끝나면 실제 접속 가능한 URL을 알려준다.

배포된 사이트도 한번 확인해보자.

(프롬프트 입력)
```
배포된 사이트를 열어서 문제가 없는지 확인해줘.
빌드 에러가 있으면 원인도 같이 봐줘.
```

이제 Shh-eat 프로젝트는
아이디어에서 시작해서,
시장 조사, 브랜드 전략, 이미지 목업, 메일 초안, 랜딩 페이지,
그리고 실제 배포 URL까지 이어졌다.

처음에는 그냥 채팅에서 시작했지만,
결국 팀원에게 공유할 수 있는 실제 결과물까지 만든 셈이다.


## 마무리

오늘 해본 건 Codex로 일하는 가장 기본적인 흐름이다.

아이디어를 정리하고, 프로젝트를 만들고, 문서를 쌓고,
이미지를 만들고, 메일 초안을 쓰고, 웹사이트를 만들고, 배포까지 해봤다.

Codex를 더 잘 활용하려면 앞으로 알아두면 좋은 것들이 더 있다.
skills, subagents, MCP, 플러그인, 자동화, context 관리 같은 것들.

이런 내용들도 앞으로 따로 다뤄볼 예정이다.

혼자 공부해보고 싶다면 `/openai-docs` 스킬을 써서
Codex 공식 문서를 기반으로 질문해보는 것도 좋다.

컴퓨터로 할 수 있는 일이라면 일단 Codex에게 시켜볼 수 있다.
작게 시작해서, 실제 내 업무 흐름에 붙여보는 게 제일 빠른 학습 방법이다.
