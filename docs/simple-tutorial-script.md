# Codex 입문 1편 촬영 스크립트

## 영상 방향

- 목표: Codex를 처음 보는 사람이 "아, 이건 그냥 챗봇이 아니라 내 컴퓨터 위에서 일하는 도구구나"라고 이해하게 만들기.
- 톤: 캐주얼, 약간 위트 있게, 하지만 장난이 기능 설명을 덮지 않게.
- 핵심 흐름: 채팅에서 아이디어를 시작하고, 프로젝트에서 파일로 남기고, GitHub로 저장하고, 웹 리서치와 랜딩 페이지까지 이어간다.
- 이번 편에서 깊게 다루지 않을 것: Skills, Subagents, MCP 세부 구조, 자동화 심화. 마지막에 다음 편 예고로만 언급한다.

## 표기

- `[SLIDE]`: 슬라이드로 보여줄 구간
- `[DEMO]`: 실제 Codex 화면에서 보여줄 구간
- `[TALK]`: 말로 설명하는 구간
- `[PROMPT]`: 화면에 입력할 프롬프트
- `[CUT]`: 편집으로 짧게 넘겨도 되는 구간

# 0. 오프닝

[SLIDE]

제목:

`Codex로 아이디어 하나를 실제 프로젝트로 밀어붙이기`

[TALK]

오늘은 Codex를 처음 써보는 분들 기준으로,
"그래서 이걸로 뭘 할 수 있는데?"를 한번 보여드리려고 합니다.

짧게 말하면 Codex는 개발자용 ChatGPT라기보다는,
내 컴퓨터에서 파일을 만들고, 코드를 고치고, 브라우저로 확인하고,
필요하면 GitHub까지 연결해서 일을 이어가는 AI 작업자에 가깝습니다.

그냥 설명만 하면 약간 제품 소개 같은 느낌이 나니까,
오늘은 가상의 커머스 브랜드를 하나 만들어보면서 진행해볼게요.

아이템은 이겁니다.

회의 중에도 눈치 안 보고 먹을 수 있는 저소음 오피스 스낵.

바삭 소리 적고, 포장 뜯는 소리 적고, 손에 가루 덜 묻는 간식.
이름은 대충 `Shh-eat`.
먹긴 먹는데 조용히 먹자. 약간 회사원의 생존 식량입니다.

오늘 목표는 이 아이디어를 그냥 채팅 기록으로 흘려보내지 않고,
실제 프로젝트 폴더 안에 문서로 남기고,
GitHub에 저장하고,
웹에서 시장 조사도 하고,
마지막에는 랜딩 페이지 만드는 흐름까지 보는 겁니다.

# 1. Codex를 언제 쓰는가

[SLIDE]

제목:

`ChatGPT처럼 쓰되, 결과물은 파일로 남긴다`

내용:

- 가벼운 질문: 채팅
- 파일이 생기는 작업: 프로젝트
- 코드, 문서, 이미지, 웹페이지: 프로젝트에서 관리
- 작업 이력: GitHub로 저장

[TALK]

Codex를 너무 어렵게 생각할 필요는 없습니다.

가벼운 질문은 그냥 채팅에서 하면 됩니다.
"이 문장 좀 다듬어줘", "아이디어 좀 내줘", "이 개념 설명해줘" 같은 건 채팅으로 충분합니다.

근데 우리가 실제로 일을 한다는 건,
대부분 뭔가가 남아야 하잖아요.

문서가 남거나,
코드가 남거나,
이미지가 남거나,
웹사이트가 남거나.

채팅창에서 아무리 좋은 얘기를 해도,
결국 "그거 어디 있어요?"라고 물어봤을 때
"어... 저번 대화 중간쯤에 있습니다" 이러면 바로 업무력 30점 깎입니다.

그래서 파일이 생기는 작업은 프로젝트에서 시작하는 게 좋습니다.
오늘도 처음에는 채팅으로 아이데이션을 해보고,
바로 프로젝트로 넘어가서 결과물을 파일로 쌓아보겠습니다.

# 3. 채팅에서 아이데이션 시작하기

[SLIDE]

제목:

`먼저 채팅에서 가볍게 던져보기`

내용:

- 아이디어를 빠르게 펼칠 때 좋음
- 방향성을 잡는 데 좋음
- 하지만 파일로 남는 업무에는 한계가 있음

[TALK]

이제 가상의 상황을 하나 만들겠습니다.

팀장님이 갑자기 오셔서 말합니다.

"우리 새로운 커머스 브랜드 하나 해보자."

이런 문장은 대체로 평온하게 들리지만,
실무자 입장에서는 오늘 일정이 끝났다는 뜻입니다.

아이템은 회의 중에도 눈치 안 보고 먹을 수 있는
저소음 오피스 스낵이라고 해볼게요.

먼저 채팅에서 아이데이션을 해봅니다.

[DEMO]

새 채팅을 열고 아래 프롬프트를 입력한다.

[PROMPT]

```text
우리는 새로운 커머스 브랜드를 런칭해야 하는 팀이야.

아이템은 회의 중에도 눈치 안 보이게 먹을 수 있는 "저소음 오피스 스낵"이야.
바삭거리는 소리, 포장 뜯는 소리, 손에 묻는 가루 같은 불편함을 줄인 직장인용 간식이라는 컨셉이야.

이 아이템을 바탕으로 커머스 스타트업에서 하는 것처럼 아이데이션해보자.
```

[TALK]

결과를 보면 브랜드 이름, 타깃, 고객 문제, 제품 아이디어 같은 것들이 나올 겁니다.

여기서 마음에 드는 이름이 나오면 잡고 가면 됩니다.
저는 이 브랜드 이름을 `Shh-eat`으로 하겠습니다.

`Shh` 조용히 하라는 뜻이고,
`eat` 먹는다는 뜻이니까,
합치면 거의 직장인의 은밀한 칼로리 작전입니다.

[DEMO]

결과 중 마음에 드는 내용을 짚는다.

[TALK]

여기까지는 채팅이 아주 좋습니다.
빠르게 아이디어를 펼칠 수 있으니까요.

근데 문제가 있습니다.
이걸 업무로 이어가려면 문서가 필요합니다.

팀원한테 보여주거나,
내일 다시 이어서 작업하거나,
랜딩 페이지 만들 때 참고하려면
채팅 기록 어딘가에 숨어 있으면 안 됩니다.

그래서 이제 프로젝트를 만들 겁니다.

가기 전에 현재 아이데이션을 새 프로젝트에서 이어갈 수 있게 정리해달라고 해보겠습니다.

[PROMPT]

```text
지금까지 아이데이션한 내용을 새 프로젝트에서 바로 이어갈 수 있게 간결하게 정리해줘.
브랜드명은 Shh-eat으로 잡고,
타깃, 고객 문제, 제품 컨셉, 차별화 포인트 중심으로 정리해줘.
```

[CUT]

응답 생성 과정은 빠르게 넘긴다.

# 4. 프로젝트 만들기

[SLIDE]

제목:

`프로젝트는 작업 폴더다`

내용:

- 문서와 코드가 실제 파일로 저장됨
- 새 스레드를 열어도 프로젝트 맥락을 유지할 수 있음
- Finder에서 파일 위치를 확인할 수 있음

[TALK]

이제 `Shh-eat` 프로젝트를 만들어보겠습니다.

프로젝트 이름은 가능하면 영어로, 공백 없이 만드는 걸 추천합니다.
나중에 코드나 터미널에서 다룰 때 편합니다.

한국어 프로젝트명도 가능은 한데,
가끔 컴퓨터가 사람 마음을 그렇게 넓게 이해하지 않습니다.

[DEMO]

Codex에서 새 프로젝트 생성.
프로젝트 이름은 `shh-eat`.

[TALK]

프로젝트가 생성되면,
이 안에서 만들어지는 문서나 코드는 내 컴퓨터의 실제 폴더에 저장됩니다.

어디 저장되는지 궁금하면 프로젝트에서 우클릭해서 Finder로 열어보면 됩니다.

이게 중요합니다.
Codex가 그냥 말만 하는 게 아니라,
내 컴퓨터 안의 실제 파일을 다루고 있다는 뜻입니다.

[DEMO]

프로젝트 폴더를 Finder에서 여는 장면.

# 5. AGENTS.md 만들기

[SLIDE]

제목:

`AGENTS.md: 프로젝트 설명서`

내용:

- 이 프로젝트가 무엇인지
- 어떤 방향으로 작업해야 하는지
- 매번 반복해서 설명하지 않아도 되는 기본 맥락

[TALK]

프로젝트를 만들었으면 제일 먼저 할 일은
`AGENTS.md`를 만드는 겁니다.

이 파일은 쉽게 말하면 Codex에게 주는 프로젝트 설명서입니다.

우리 프로젝트가 어떤 프로젝트인지,
어떤 톤으로 작업해야 하는지,
뭘 중요하게 봐야 하는지 적어두는 곳입니다.

매번 새 스레드를 열 때마다
"안녕, 이 프로젝트는 저소음 오피스 스낵 브랜드고..."를 반복하면
사람도 지치고 AI도 눈치 봅니다.

`AGENTS.md`에 적어두면,
프로젝트 안의 스레드들이 그 맥락을 기본으로 가지고 시작합니다.

[DEMO]

프로젝트 스레드에서 아래 프롬프트 입력.

[PROMPT]

```text
아래 아이데이션 내용을 바탕으로 현재 프로젝트에 AGENTS.md를 만들어줘.

요구사항:
- Shh-eat이 어떤 프로젝트인지 간결하게 설명
- 브랜드 방향, 타깃, 제품 컨셉 포함
- 나중에 랜딩 페이지까지 만들 예정이라는 점 포함
- 너무 길지 않게 작성

---

[여기에 이전 채팅에서 정리한 아이데이션 내용을 붙여넣기]
```

[CUT]

파일 생성 과정은 빠르게 넘긴다.

[DEMO]

생성된 `AGENTS.md`를 열어 보여준다.

[TALK]

이제 이 프로젝트에는 기본 맥락이 생겼습니다.

여기서 중요한 건 `AGENTS.md`를 완벽하게 쓰려고 너무 오래 붙잡고 있을 필요가 없다는 겁니다.

처음에는 짧게 만들고,
프로젝트 방향이 바뀌면 계속 업데이트하면 됩니다.

회사에서 프로젝트 킥오프 문서도 처음부터 완벽한 경우 거의 없잖아요.
보통은 "일단 시작합시다"라고 하고,
3일 뒤에 전혀 다른 프로젝트가 되어 있습니다.

그럴 때 `AGENTS.md`도 같이 바꿔주면 됩니다.

[PROMPT]

```text
아이데이션과 기획이 끝나면 나중에 랜딩 페이지까지 만들 예정이야.
현재 AGENTS.md를 최대한 간결하게 다듬어줘.
```

[TALK]

이런 식으로 계속 정리해가면 됩니다.

# 6. GitHub 연결하기

[SLIDE]

제목:

`GitHub는 작업 이력 저장소`

내용:

- 커밋: 현재 상태를 저장
- 푸시: GitHub에 업로드
- 협업과 배포의 출발점

[TALK]

이제 GitHub 얘기를 잠깐 하겠습니다.

GitHub를 아주 단순하게 말하면
개발자들이 쓰는 구글 드라이브 같은 겁니다.

조금 더 정확히는,
파일을 그냥 저장하는 게 아니라
누가, 언제, 뭘 바꿨는지 이력까지 저장합니다.

AI랑 작업할수록 이게 중요해집니다.

AI가 일을 많이 해주면 좋긴 한데,
그만큼 "어디까지 했지?", "뭐가 바뀌었지?", "이전 상태로 돌아갈 수 있나?"
이런 질문이 더 중요해집니다.

그래서 Codex랑 작업할 때 GitHub는 거의 필수에 가깝습니다.

[SLIDE]

제목:

`딱 두 단어만 기억하기`

내용:

`commit = 저장`

`push = 업로드`

[TALK]

처음에는 두 단어만 기억하면 됩니다.

커밋은 저장입니다.
"여기까지 작업한 상태 괜찮으니까 저장해줘"입니다.

푸시는 업로드입니다.
"내 컴퓨터에 저장한 작업을 GitHub에도 올려줘"입니다.

문서 작업하다가 저장 버튼 누르는 게 커밋이고,
그 파일을 팀 드라이브에 올리는 게 푸시라고 생각하면 됩니다.

[DEMO]

GitHub CLI가 없거나 로그인이 안 되어 있는 경우 새 스레드 예시만 보여준다.

[PROMPT]

```text
github cli 설치하고 gh auth login까지 해보자.
```

[TALK]

이런 환경 설정도 Codex에게 요청할 수 있습니다.

다만 계정 로그인이나 권한 승인처럼 사람이 직접 확인해야 하는 구간이 있습니다.
Codex가 모든 걸 몰래 처리하는 게 아니라,
필요한 순간에는 권한을 물어봅니다.

이건 귀찮은 게 아니라 정상입니다.
내 계정으로 외부 서비스를 건드리는 일이니까요.

[DEMO]

현재 프로젝트를 GitHub에 올리는 요청 예시.

[PROMPT]

```text
이 프로젝트를 GitHub에 올리고 싶어.
현재 폴더를 기준으로 GitHub 레포를 만들고,
초기 커밋 후 푸시까지 진행해줘.
```

[CUT]

실제 인증/대기 시간은 편집으로 정리한다.

[TALK]

앞으로는 뭔가 의미 있는 작업이 끝날 때마다 커밋하면 됩니다.

문서 만들었다, 커밋.
리서치 끝났다, 커밋.

# 7. 웹 리서치로 문서 만들기

[TALK]

지금까지는 우리가 상상한 아이디어입니다.

근데 실제 업무에서는 여기서 멈추면 위험합니다.

"회의 중에 조용히 먹는 스낵 좋지 않나요?"

이 말만 가지고 바로 제품 만들면,
그건 브랜드 기획이라기보다는 야식 먹다가 떠오른 확신에 가깝습니다.

그래서 웹에서 시장 조사를 해보겠습니다.

중요한 건 Codex에게 그냥 "시장 조사해줘"라고 하면 결과가 넓게 퍼질 수 있다는 겁니다.

조사 주제, 요구사항, 저장 위치를 같이 줘야 합니다.

[DEMO]

새 스레드에서 아래 프롬프트 입력.

[PROMPT]

```text
Shh-eat 브랜드를 더 구체화하기 위해 웹에서 시장 조사를 해줘.

조사 주제:
- 오피스 스낵 / 직장인 간식 시장 트렌드
- 회의 중 먹기 좋은 간식, 책상에서 먹기 좋은 간식 관련 니즈
- 소음이 적고 가루가 덜 묻는 간식이나 포장 사례
- 유사한 D2C 스낵 브랜드 또는 오피스 간식 구독 서비스
- Shh-eat이 차별화할 수 있는 포지션

요구사항:
- 실제 웹에서 찾은 정보만 사용해줘
- 출처 링크를 같이 남겨줘
- 한국어 자료를 우선 참고해줘
- 근거가 약한 내용은 약하다고 표시해줘
- 결과는 docs/research/market-research.md 파일로 정리해줘
```

[CUT]

검색과 문서 작성 시간은 빠르게 넘긴다.

[DEMO]

생성된 `docs/research/market-research.md`를 열어 보여준다.

[TALK]

이제 이건 그냥 대화가 아니라 프로젝트 안에 남은 자료입니다.

나중에 랜딩 페이지를 만들 때도,
브랜드 전략을 정리할 때도,
팀원에게 공유할 때도 이 문서를 참고하면 됩니다.

그리고 출처 링크를 남기라고 한 게 중요합니다.

AI가 자신감 있게 말한다고 다 근거가 있는 건 아닙니다.
사람도 회의실에서 자신감 있게 말하면 가끔 아무도 의심 안 하잖아요.
그래서 출처를 같이 남기게 해야 합니다.

# 8. 브랜드 전략 문서 만들기

방금 만든 docs/research/market-research.md 문서를 참고해서

# 10. 큰 작업은 Plan mode로 시작하기

[SLIDE]

제목:

`큰 작업은 바로 만들기 전에 계획부터`

내용:

- 새 웹앱 만들기
- 기존 파일 참고하기
- 여러 파일 수정하기
- 설치와 실행이 필요한 작업

[TALK]

이제 랜딩 페이지를 만들어보겠습니다.

여기서부터는 작업이 조금 커집니다.

Next.js 프로젝트를 셋업해야 하고,
방금 만든 문서들을 참고해야 하고,
페이지도 만들고,
로컬에서 실행해서 브라우저로 확인해야 합니다.

이런 작업은 바로 "만들어줘"라고 해도 되지만,
처음에는 Plan mode로 계획을 먼저 세우는 걸 추천합니다.

Plan mode는 말 그대로 작업 계획부터 세우는 모드입니다.
바로 파일을 막 수정하지 않고,
현재 폴더 구조를 보고 어떤 순서로 진행할지 정리합니다.

약간 공사 전에 도면 보는 느낌입니다.
물론 현실의 많은 공사는 도면을 봐도 혼란스럽지만,
그래도 안 보는 것보다는 낫습니다.

[DEMO]

Plan mode로 전환하는 화면을 보여준다.

[PROMPT]

```text
Shh-eat 랜딩 페이지를 Next.js로 만들고 싶어.
현재 shh-eat 폴더를 그대로 쓰고, git은 새로 만들지 않았으면 좋겠어.

지금까지 만든 docs 폴더의 문서들을 참고해서
작업 계획을 먼저 세워줘.

아직 파일은 수정하지 말고,
어떤 순서로 진행할지와 확인해야 할 사항을 알려줘.
```

[TALK]

그러면 Codex가 보통 이런 걸 확인합니다.

현재 프로젝트 구조,
Next.js를 새로 설치할지,
어떤 문서를 참고할지,
랜딩 페이지에 들어갈 섹션,
로컬 실행 방법,
커밋 시점.

계획을 보고 괜찮으면 진행시키면 됩니다.

[PROMPT]

```text
좋아. 이 계획대로 진행해줘.
```

# 11. 랜딩 페이지 만들고 로컬에서 확인하기

[SLIDE]

제목:

`로컬 실행과 브라우저 확인`

내용:

- 설치가 필요하면 Codex가 물어봄
- 개발 서버 실행
- 인앱 브라우저로 확인
- 수정 요청 후 반복

[TALK]

Next.js 같은 웹 프로젝트는 설치가 필요할 수 있습니다.

Codex가 `npm install` 같은 명령을 실행해야 하면 권한을 물어볼 수 있습니다.
승인하면 설치가 진행됩니다.

그리고 개발 서버를 실행할지 물어볼 텐데,
보통은 `npm run dev`로 실행합니다.

개발 서버는 내 컴퓨터에서만 보이는 임시 웹사이트를 띄우는 겁니다.
인터넷 전체에 공개되는 건 아닙니다.

혼자 작업실에 시제품 올려놓고 보는 느낌입니다.

[DEMO]

설치, `npm run dev`, 로컬 URL 확인.

[TALK]

이제 Codex는 앱 안의 브라우저로 로컬 페이지를 열어보고,
화면이 제대로 나오는지 확인할 수 있습니다.

여기서 마음에 안 드는 부분은 그냥 말로 수정 요청하면 됩니다.

[PROMPT]

```text
현재 제품 이미지가 너무 작아 보여.
Hero 섹션을 더 제품 중심으로 바꾸고,
카피는 이미지 위에 올라오게 해줘.
텍스트 가독성도 같이 확인해줘.
```

[TALK]

이런 식으로 요청하면 됩니다.

중요한 건 "예쁘게 해줘"보다
어떤 문제가 있고, 어떤 방향으로 바꾸고 싶은지 말하는 겁니다.

"예쁘게"는 사람 사이에서도 위험한 말입니다.
한 명은 애플을 생각하고,
한 명은 PC방 포스터를 생각할 수 있습니다.

[DEMO]

수정 후 브라우저 확인.

[TALK]

작업이 끝났으면 개발 서버는 꺼도 됩니다.

[PROMPT]

```text
개발 서버 종료해줘.
```

다시 보고 싶으면 이렇게 말하면 됩니다.

[PROMPT]

```text
개발 서버 다시 실행하고 인앱 브라우저에서 열어줘.
```

# 12. 배포는 짧게 소개

[SLIDE]

제목:

`로컬에서 인터넷으로`

내용:

- 로컬: 내 컴퓨터에서만 보임
- 배포: 다른 사람도 URL로 접속 가능
- Next.js는 Vercel과 궁합이 좋음
- GitHub에 푸시된 코드가 배포의 기준이 됨

[TALK]

랜딩 페이지가 로컬에서 잘 보이면,
이제 다른 사람도 볼 수 있게 배포할 수 있습니다.

배포는 내 컴퓨터에서만 보이던 웹사이트를
실제 인터넷 주소로 올리는 겁니다.

Next.js 프로젝트는 보통 Vercel에 배포하기 좋습니다.
GitHub 레포와 연결해두면,
앞으로 푸시할 때마다 자동으로 새 버전을 배포할 수도 있습니다.

이 부분은 계정 연결과 서비스 권한이 들어가서,
환경마다 화면이 조금 다를 수 있습니다.

그래서 오늘은 개념과 요청 방식만 보여드리겠습니다.

[DEMO]

배포 요청 예시만 보여준다.

[PROMPT]

```text
이 Shh-eat 랜딩 페이지를 Vercel에 배포하고 싶어.
GitHub에 올라가 있는 현재 레포를 사용하면 되고,
배포가 끝나면 접속 가능한 URL을 알려줘.
```

[TALK]

배포 후에는 꼭 확인을 요청하는 게 좋습니다.

[PROMPT]

```text
배포된 사이트를 열어서 문제가 없는지 확인해줘.
빌드 에러가 있으면 원인도 같이 봐줘.
```

[TALK]

AI가 만들었으니까 괜찮겠지, 하고 넘어가면 안 됩니다.

우리는 언제나 봐야 합니다.
화면이 깨졌는지,
버튼이 눌리는지,
모바일에서 이상하지 않은지.

Codex가 확인을 도와줄 수는 있지만,
최종 판단은 작업하는 사람이 해야 합니다.
