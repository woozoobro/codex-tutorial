# 6. GitHub 연결하기

[SLIDE: GitHub가 필요한 이유]

이제 GitHub 얘기를 잠깐 하겠습니다.

GitHub는 아주 단순하게 말하면
개발자들이 쓰는 구글 드라이브 같은 겁니다.

조금 더 정확히는,
파일을 그냥 저장하는 게 아니라
누가, 언제, 뭘 바꿨는지 이력까지 저장합니다.

AI랑 작업할수록 이게 중요해집니다.

AI가 일을 많이 해주면 좋긴 한데,
그만큼 "어디까지 했지?", "뭐가 바뀌었지?", "이전 상태로 돌아갈 수 있나?"
이런 질문이 더 중요해집니다.

그래서 Codex랑 작업할 때 GitHub가 거의 필수에 가깝습니다.

[SLIDE: 커밋과 푸시]
처음에는 두 단어만 기억하면 됩니다.

커밋은 저장입니다.
"여기까지 작업한 상태 괜찮으니까 저장해줘"입니다.

푸시는 업로드입니다.
"내 컴퓨터에 저장한 작업을 GitHub에도 올려줘"입니다.

문서 작업하다가 저장 버튼 누르는 게 커밋이고,
그런 작업 내역들을 드라이브에 올리는 게 푸시라고 생각하면 됩니다.

일단 깃헙 계정을 하나 만들어주셔야 합니다.

아마 경우에 따라서 막 엄청 어려운 퍼즐들을 풀라고 할때가 있는데.
이게 사람인지 ai인지 검증을 하는거라. 꽤 빡세게 나오더라고요.
만약에 퍼즐이 나오지 않았다면 선택받으신 분...
빠르게 회원가입 하시면 됩니다.

[DEMO]
이런 환경 설정도 Codex한테 요청할 수 있어요.
새로운 스레드에서

# 15. Vercel로 배포하기

[SLIDE: 배포]

랜딩 페이지가 로컬에서 잘 보이면,
이제 다른 사람도 볼 수 있게 배포할 수 있습니다.

배포는 내 컴퓨터에서만 보이던 웹사이트를 실제 인터넷 주소로 올리는 겁니다.

Next.js 프로젝트는 보통 Vercel에 배포하기 좋습니다.
GitHub 레포와 연결해두면, 앞으로 푸시할 때마다 자동으로 새 버전을 배포할 수도 있어요.

먼저 Vercel 계정이 필요한데요.
보통은 GitHub 계정으로 로그인하면 편합니다.

그리고 Vercel이 내 GitHub 레포에 접근할 수 있도록 권한을 승인해야 합니다.

[DEMO]

Codex에서 플러그인 화면을 열고 Vercel 플러그인을 검색하는 장면.

그 다음 Codex에서 Vercel 플러그인을 추가합니다.

처음 사용할 때는 Vercel 계정 연결이나 권한 승인이 필요할 수 있습니다.

[DEMO]

새 스레드에서 Vercel을 멘션한다.

[PROMPT]

```text
@vercel
이 Shh-eat 랜딩 페이지를 Vercel에 배포하고 싶어.
GitHub에 올라가 있는 shh-eat 레포를 사용하면 되고,
배포가 끝나면 접속 가능한 URL을 알려줘.
```

이렇게 요청하면 Codex가 Vercel 배포에 필요한 설정을 확인하고,
배포가 끝나면 실제 접속 가능한 URL을 알려줍니다.

배포 후에는 꼭 확인을 요청하는 게 좋습니다.

[PROMPT]

```text
배포된 사이트를 열어서 문제가 없는지 확인해줘.
빌드 에러가 있으면 원인도 같이 봐줘.
```

커밋하고 푸시해줘.
